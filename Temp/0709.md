下面按“你会 JS/TS”的视角来整理。核心目标不是死记 Python / LangChain 名词，而是把它们映射到你已经熟悉的前端/Node 思维里。

## 一、Python 最小基础

### 1. 变量

Python 变量和 JS 很像，都是动态绑定。

JS/TS：

```
let name = "Tom"
let age = 18
let active = true
```

Python：

```
name = "Tom"
age = 18
active = True
```

差异：

```
# Python 没有 let / const
# 布尔值是 True / False，首字母大写
# 空值是 None，不是 null / undefined
```

JS/TS：

```
let value = null
```

Python：

```
value = None
```

你可以这样理解：

```
Python 变量 ≈ JS 的 let
Python None ≈ JS 的 null
```

---

### 2. 函数

JS/TS：

```
function add(a: number, b: number): number {
  return a + b
}
```

Python：

```
def add(a: int, b: int) -> int:
    return a + b
```

区别：

- Python 用 `def` 定义函数。
- Python 用缩进表示代码块。
- Python 类型标注不是强制的，但推荐写。
- `-> int` 表示返回值类型。

匿名函数对比：

JS：

```
const double = (x: number) => x * 2
```

Python：

```
double = lambda x: x * 2
```

但 Python 里复杂逻辑一般不用 `lambda`，直接 `def`。

---

### 3. 类

JS/TS：

```
class User {
  name: string

  constructor(name: string) {
    this.name = name
  }

  sayHi() {
    return `Hi, ${this.name}`
  }
}
```

Python：

```
class User:
    def __init__(self, name: str):
        self.name = name

    def say_hi(self) -> str:
        return f"Hi, {self.name}"
```

重点映射：

```
constructor ≈ __init__
this ≈ self
```

Python 里方法第一个参数通常是 `self`，相当于 JS 里的 `this`，但需要显式写出来。

使用：

```
user = User("Tom")
print(user.say_hi())
```

---

### 4. list / dict

#### list

JS/TS 数组：

```
const arr = [1, 2, 3]
arr.push(4)
```

Python list：

```
arr = [1, 2, 3]
arr.append(4)
```

常用操作：

```
arr[0]      # 第一个元素
arr[-1]     # 最后一个元素
len(arr)    # 长度
```

JS：

```
arr.length
```

Python：

```
len(arr)
```

#### dict

JS 对象：

```
const user = {
  name: "Tom",
  age: 18
}

console.log(user.name)
console.log(user["age"])
```

Python dict：

```
user = {
    "name": "Tom",
    "age": 18
}

print(user["name"])
print(user.get("age"))
```

区别：

Python dict 的 key 通常用字符串，访问时更常用：

```
user["name"]
```

如果 key 不存在，`user["name"]` 会报错。更安全的是：

```
user.get("name")
```

你可以理解：

```
Python list ≈ JS array
Python dict ≈ JS object / Map
```

---

### 5. 类型标注

TS：

```
function greet(name: string): string {
  return `Hello ${name}`
}
```

Python：

```
def greet(name: str) -> str:
    return f"Hello {name}"
```

list 类型：

TS：

```
const names: string[] = ["Tom", "Jack"]
```

Python：

```
names: list[str] = ["Tom", "Jack"]
```

dict 类型：

TS：

```
const scores: Record<string, number> = {
  Tom: 90
}
```

Python：

```
scores: dict[str, int] = {
    "Tom": 90
}
```

复杂对象在 TS 里用 interface：

```
interface User {
  name: string
  age: number
}
```

Python 常用 Pydantic 或 dataclass，FastAPI 里重点是 Pydantic：

```
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
```

你可以理解：

```
Python 类型标注 ≈ TS 类型提示
Pydantic Model ≈ TS interface + 运行时校验
```

这是重点：TS 类型主要在编译期，Pydantic 会在运行时校验数据。

### 6. 文件读写

Node.js：

```
import fs from "fs"

const text = fs.readFileSync("a.txt", "utf-8")
fs.writeFileSync("b.txt", text)
```

Python：

```
with open("a.txt", "r", encoding="utf-8") as f:
    text = f.read()

with open("b.txt", "w", encoding="utf-8") as f:
    f.write(text)
```

重点是 `with open(...) as f`。

你可以理解：

```
with ≈ 自动帮你 close 文件
```

读 JSON：

JS：

```
const obj = JSON.parse(text)
```

Python：

```
import json

obj = json.loads(text)
```

写 JSON：

```
import json

with open("data.json", "w", encoding="utf-8") as f:
    json.dump(obj, f, ensure_ascii=False, indent=2)
```

```
npm install
package.json
node_modules
```

Python 里对应：

```
pip install
requirements.txt
venv
```

类比：

```
npm ≈ pip
package.json ≈ requirements.txt / pyproject.toml
node_modules ≈ venv 里的 site-packages
```

创建虚拟环境：

```
python -m venv .venv
```

激活：

Windows PowerShell：

```
.venv\Scripts\Activate.ps1
```

安装依赖：

```
pip install fastapi uvicorn
```

导出依赖：

```
pip freeze > requirements.txt
```

```
pip install -r requirements.txt
```

你可以理解：

```
venv 是 Python 项目的独立依赖环境，避免全局污染。
```

类似每个 Node 项目都有自己的 `node_modules`。

### 8. async / await

JS/TS：

```
async function getData() {
  const res = await fetch("/api/data")
  return await res.json()
}
```

Python：

```
import httpx

async def get_data():
    async with httpx.AsyncClient() as client:
        res = await client.get("https://example.com")
        return res.json()
```

概念基本一样：

```
async 表示异步函数
await 等待异步结果
```

FastAPI 中常见：

```
@app.get("/users")
async def get_users():
    return {"users": []}
```

注意：

如果你在 async 函数里调用阻塞代码，比如大文件处理、同步数据库、同步 HTTP 请求，会影响性能。

简单理解：

```
Node 天生大量异步
Python 既可以同步，也可以异步
FastAPI 支持 async，但你要确保里面用的库也是异步友好的
```

## 二、FastAPI 最小基础

你可以把 FastAPI 理解为：

```
FastAPI ≈ Express / NestJS
Pydantic ≈ DTO + interface + runtime validation
uvicorn ≈ Node HTTP server
```

### 1. 路由

Express：

```
app.get("/health", (req, res) => {
  res.json({ ok: true })
})
```

FastAPI：

```
from fastapi import FastAPI

app = FastAPI()

@app.get("/health")
def health():
    return {"ok": True}
```

启动：

```
uvicorn main:app --reload
```

意思是：

```
main:app
main 是 main.py 文件
app 是 FastAPI 实例
```

---

### 2. 请求参数

#### Query 参数

请求：

```
GET /users?name=Tom
```

FastAPI：

```
@app.get("/users")
def get_users(name: str):
    return {"name": name}
```

Express：

```
app.get("/users", (req, res) => {
  const name = req.query.name
})
```

---

#### Path 参数

请求：

```
GET /users/123
```

FastAPI：

```
@app.get("/users/{user_id}")
def get_user(user_id: int):
    return {"user_id": user_id}
```

Express：

```
app.get("/users/:userId", (req, res) => {
  const userId = req.params.userId
})
```

---

#### Body 参数

TS DTO：

```
interface CreateUserDto {
  name: string
  age: number
}
```

FastAPI + Pydantic：

```
from pydantic import BaseModel

class CreateUserRequest(BaseModel):
    name: str
    age: int

@app.post("/users")
def create_user(data: CreateUserRequest):
    return data
```

FastAPI 会自动：

- 解析 JSON body
- 校验字段
- 生成接口文档
- 类型错误时报 422

这比 Express 原生强很多，更接近 NestJS。

---

### 3. 响应

直接返回 dict：

```
@app.get("/hello")
def hello():
    return {"message": "hello"}
```

FastAPI 会自动转 JSON。

指定响应模型：

```
class UserResponse(BaseModel):
    id: int
    name: str

@app.get("/users/{user_id}", response_model=UserResponse)
def get_user(user_id: int):
    return {
        "id": user_id,
        "name": "Tom",
        "password": "secret"
    }
```

FastAPI 会根据 `response_model` 过滤字段，`password` 不会返回。

这点很有用。

## 三、Pydantic 模型

Pydantic 是你做 FastAPI 必须理解的。

你可以把它理解为：

```
Pydantic = TS interface + class-validator + 数据转换
```

例子：

```
from pydantic import BaseModel

class ChatRequest(BaseModel):
    message: str
    temperature: float = 0.7
```

请求：

```
{
  "message": "你好"
}
```

FastAPI 会得到：

```
data.message
data.temperature
```

其中 `temperature` 自动使用默认值 `0.7`。

如果传错：

```
{
  "message": 123
}
```

Pydantic 会校验并报错。

更复杂：

```
class Message(BaseModel):
    role: str
    content: str

class ChatRequest(BaseModel):
    messages: list[Message]
```

对应 TS：

```
interface Message {
  role: string
  content: string
}

interface ChatRequest {
  messages: Message[]
}
```

## 四、LangChain / RAG 最小基础

先说一句关键的：

```
LangChain 不是大模型。
LangChain 是帮你组织 Prompt、文档、检索、工具调用、流程编排的开发框架。
```

你可以类比：

```
Vue 不是浏览器，Vue 是组织前端应用的框架。
LangChain 不是 AI，LangChain 是组织 AI 应用的框架。
```

---

## 1. PromptTemplate

普通写法：

```
prompt = f"请根据以下资料回答问题：{context}，问题是：{question}"
```

PromptTemplate 写法：

```
from langchain_core.prompts import PromptTemplate

template = PromptTemplate.from_template("""
请根据以下资料回答问题：

资料：
{context}

问题：
{question}
""")

prompt = template.format(
    context="产品支持一年质保",
    question="质保多久？"
)
```

类比 TS：

```
const template = `
请根据以下资料回答问题：

资料：
${context}

问题：
${question}
`
```

PromptTemplate 的价值：

- 统一管理 Prompt
- 避免字符串拼接混乱
- 方便复用
- 方便后续做版本管理

---

## 2. Document

LangChain 里的 `Document` 就是“带元数据的文本”。

普通字符串：

```
text = "产品支持一年质保"
```

Document：

```
from langchain_core.documents import Document

doc = Document(
    page_content="产品支持一年质保",
    metadata={
        "source": "售后手册.pdf",
        "page": 3
    }
)
```

你可以理解：

```
interface Document {
  pageContent: string
  metadata: {
    source?: string
    page?: number
  }
}
```

为什么不用普通字符串？

因为 RAG 需要知道答案来源。

例如模型回答：

```
产品支持一年质保。
来源：售后手册.pdf 第 3 页
```

这个来源就来自 `metadata`。

---

## 3. TextSplitter

TextSplitter 是文本切分器。

为什么需要切分？

假设一本 PDF 有 10 万字，不能全部塞给模型，所以要切成小段。

```
大文档
-> chunk 1
-> chunk 2
-> chunk 3
```

LangChain 示例：

```
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)

chunks = splitter.split_documents(docs)
```

参数解释：

```
chunk_size = 每段多长
chunk_overlap = 相邻段落重叠多少
```

为什么要 overlap？

因为有些句子上下文跨段，如果完全切断，语义会断。

类比前端分页：

```
一篇长文不能一次展示，就分页。
但分页时为了上下文，前后页可能保留一点重复内容。
```

面试常问：

```
chunk 太大：检索不准，塞进模型成本高
chunk 太小：上下文不完整，回答容易缺信息
```

## 4. Embedding

Embedding 是把文本变成向量。

例如：

```
"一年质保" -> [0.12, -0.98, 0.33, ...]
"保修期多久" -> [0.11, -0.95, 0.31, ...]
```

语义相近的文本，向量距离更近。

你可以理解：

```
Embedding = 文本的语义坐标
```

不是关键词匹配。

关键词匹配：

```
问题：保修多久？
文档：一年质保
```

关键词可能匹配不到“保修”和“质保”的关系。

Embedding 可以知道它们语义接近。

在 RAG 里：

```
用户问题 -> embedding
文档 chunk -> embedding
比较相似度 -> 找到最相关 chunk
```

---

## 5. VectorStore

VectorStore 是存向量的数据库。

你可以类比：

```
普通数据库：根据 id、name、time 查询
向量数据库：根据语义相似度查询
```

常见：

- pgvector
- Milvus
- Chroma
- FAISS

普通数据库查询：

```
select * from docs where title like '%质保%'
```

向量数据库查询：

```
找出和“保修多久”语义最接近的 5 个文档片段
```

在 LangChain 里：

```
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embedding_model
)
```

实际企业项目中，你可以优先用：

```
PostgreSQL + pgvector
```

因为传统企业更容易接受 PostgreSQL，不一定愿意维护单独的向量数据库。

---

## 6. Retriever

Retriever 是“检索器”。

你可以理解：

```
VectorStore 是仓库
Retriever 是查询员
```

例子：

```
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5}
)

docs = retriever.invoke("质保多久？")
```

意思是：

```
从向量库里找出最相关的 5 个文档片段
```

RAG 流程里：

```
用户问题
-> Retriever 找资料
-> 把资料塞进 Prompt
-> LLM 生成回答
```

---

## 7. Chain

Chain 是“把多个步骤串起来”。

最简单的 RAG Chain：

```
用户问题
-> 检索文档
-> 组装 Prompt
-> 调用模型
-> 输出答案
```

类比前端/Node：

```
const docs = await retriever.search(question)
const prompt = buildPrompt(docs, question)
const answer = await llm.chat(prompt)
return answer
```

LangChain 里把这些步骤封装成 Chain。

你可以理解：

```
Chain ≈ pipeline / middleware / workflow
```

以前你在 Node 里手写流程：

```
validateInput()
queryDatabase()
callService()
formatResponse()
```

LangChain 里就是把 AI 相关步骤用统一结构串起来。

## 五、RAG 整体流程

你可以先记住这张流程：

```
离线阶段：
文档上传
-> 文档解析
-> 文本切分
-> 生成 Embedding
-> 存入 VectorStore

在线阶段：
用户提问
-> 问题生成 Embedding
-> Retriever 检索相关 chunks
-> PromptTemplate 组装上下文
-> LLM 生成回答
-> 返回答案和引用来源
```

对应概念：

```
Document：文档片段
TextSplitter：切文档
Embedding：文本转向量
VectorStore：存向量
Retriever：查相关资料
PromptTemplate：组装提示词
Chain：串起整个流程
```
